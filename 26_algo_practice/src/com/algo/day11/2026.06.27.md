## 문제 1. 부분수열의 합 2

### 문제 요약

- N개의 정수 수열에서, 크기가 양수인 **부분수열(부분집합)** 중 원소 합이 S가 되는 경우의 수를 구한다.
- "부분수열"은 연속 구간이 아니라 임의의 부분집합이다. (순서 무관, 원소를 골라 더함)
- 제한: 1 ≤ N ≤ 40, |S| ≤ 1,000,000, 각 정수의 절댓값 ≤ 100,000

---

### 내 풀이 분석

- **풀이 결과**: 실패 (문제 정의를 잘못 해석)

**알고리즘**

누적합 `sum[i]`을 만든 뒤 `sum[i] == S + sum[j]` (j < i)인 쌍의 개수를 세었다.
이는 **연속 부분배열(구간) 합 = S** 의 개수를 세는 전형적인 풀이다.

**시간복잡도**

- 이중 루프로 O(N²). (N≤40이라 속도 자체는 문제 없음)

**❌ 문제점 / 개선점**

- 가장 큰 문제: 이 문제는 **연속 구간이 아니라 부분집합**의 합을 센다.
  누적합 차이는 연속 구간만 표현하므로 비연속 부분집합(예: 1번·3번·4번 원소)을 셀 수 없다.
- 예제(`-7 -3 -2 5 8`, S=0)에서 정답이 1로 우연히 일치한다.
  연속 구간 `[-3,-2,5]` 하나만 잡히는데, 이 케이스엔 비연속 해가 없어서 결과가 같았을 뿐이다.
  비연속 해가 있는 입력에서는 즉시 틀린다.
- N≤40이라는 제한은 "2^40은 못 풀지만 2^20 두 번은 가능"하라는 **중간에서 만나기(MITM)** 시그널이다.

---

### 모범 답안: 중간에서 만나기 (Meet in the Middle)

**알고리즘**

1. 수열을 앞 절반 / 뒤 절반으로 나눈다.
2. 각 절반에 대해 모든 부분집합 합(공집합 포함, 2^(N/2)개)을 생성한다.
3. 한쪽 배열을 정렬한다.
4. 다른 쪽의 각 합 `x`에 대해, 정렬된 쪽에서 값이 `S - x`인 원소 개수를
   이분탐색(upperBound − lowerBound)으로 세어 누적한다.
5. S = 0이면 "양쪽 모두 공집합"인 빈 부분수열 1개가 포함되므로 마지막에 1을 뺀다.

**예시 추적** (`-7 -3 -2 5 8`, S=0)

```
앞 절반 {-7,-3}      → 부분합: 0, -7, -3, -10
뒤 절반 {-2,5,8}     → 부분합: 0, -2, 5, 8, 3, 6, 13, 11
각 x에 대해 (0 - x) 개수 합산 → 1 (= {-3,-2,5})
S=0이라 빈 집합 1 제거는 그 1을 더한 뒤 보정 → 최종 1
```

**시간복잡도**

| 단계 | 복잡도 |
|------|--------|
| 부분집합 합 생성 | O(2^(N/2) · N/2) |
| 정렬 | O(2^(N/2) · N/2) |
| 이분탐색 카운팅 | O(2^(N/2) · log 2^(N/2)) |
| 합계 | O(2^(N/2) · N) |

**공간복잡도**: O(2^(N/2))

---

### 핵심 정리

- **N ≤ 40 (또는 30~40대)** + "부분집합/조합 개수" → **중간에서 만나기(MITM)** 를 먼저 의심하라.
  절반씩 2^(N/2)로 쪼개고, 한쪽 정렬 후 다른 쪽에서 `target - x`를 이분탐색.
- **"부분수열" ≠ "연속 부분배열"**. 누적합/투포인터는 연속 구간 전용이다.
  문제에서 "원소를 골라 더한다"는 표현이면 부분집합 문제로 본다.
- 합 = 0 같은 경계에서는 **공집합 포함 여부**를 반드시 보정한다(크기 양수 조건).
- 정확히 같은 값의 개수를 셀 때는 `upperBound - lowerBound` 패턴이 깔끔하다.

## 문제 2. 반도체 설계

### 문제 요약

- 왼쪽 n개 포트와 오른쪽 n개 포트의 연결 관계가 주어진다.
- 연결선이 서로 **꼬이지(교차하지) 않도록** 하면서 최대 몇 개를 연결할 수 있는지 구한다.
- 왼쪽 포트는 1, 2, …, n 순서로 고정. 입력은 각 왼쪽 포트가 연결될 오른쪽 포트 번호.
- 제한: 1 ≤ n ≤ 40,000, 오른쪽 포트 번호는 1~n의 서로 다른 수.

---

### 내 풀이 분석

- **풀이 결과**: 실패 (미완성 — 입력만 읽고 종료)

**알고리즘**

"선이 꼬이지 않으려면 한 방향(증가)이 유지되어야 한다"까지는 정확히 관찰했다.
다만 이를 알고리즘으로 정식화하지 못하고 멈췄다. (주석: "감도 안잡히네", "DP 아닌 것 같은데")

**시간복잡도**

- 미완성으로 분석 불가.

**✅ 잘 짚은 것**

- "증가하는 부분 수열을 유지해야 꼬이지 않는다"는 핵심 통찰. 이게 사실상 문제의 절반이다.

**❌ 문제점 / 개선점**

- 관찰을 **LIS(최장 증가 부분 수열)** 라는 표준 문제로 연결하지 못함.
- "DP 아닌 것 같다"는 직관은 절반만 맞다. LIS는 DP로 정의되지만, n=40,000에서는
  O(n²) DP가 위험하므로 **tails 배열 + 이분탐색**의 O(n log n) 버전을 써야 한다.

---

### 모범 답안: 최장 증가 부분 수열 (LIS, O(n log n))

**알고리즘**

1. 왼쪽 포트가 1..n로 고정이므로, 꼬이지 않는 연결 = 오른쪽 포트 번호가 증가하는 부분 수열.
2. 따라서 dest 배열의 **LIS 길이**가 답이다.
3. `tails[k]` = 길이 k+1인 증가 수열이 가질 수 있는 최소 끝값. 각 원소 x마다
   `lowerBound`로 들어갈 위치를 찾아 교체(끝이면 길이 +1).

**예시 추적** (`4 2 6 3 1 5`)

```
4 → tails=[4]          (len 1)
2 → tails=[2]          (4를 2로 교체)
6 → tails=[2,6]        (len 2)
3 → tails=[2,3]        (6을 3으로 교체)
1 → tails=[1,3]        (2를 1로 교체)
5 → tails=[1,3,5]      (len 3)
=> 답 3
```

**시간복잡도**

| 단계 | 복잡도 |
|------|--------|
| 각 원소마다 이분탐색 | O(log n) |
| 합계 | O(n log n) |

**공간복잡도**: O(n)

---

### 핵심 정리

- "두 열의 연결선이 교차하지 않게 최대 매칭" 류는 한쪽을 정렬/고정한 뒤 **LIS**로 환원된다.
- n이 수만 단위면 O(n²) DP LIS는 위험 → **tails + 이분탐색** O(n log n) 버전을 반사적으로 떠올리자.
- `tails` 배열의 값 자체는 실제 LIS가 아니지만, **길이는 정확**하다(이게 핵심).
- 증가/감소·strict/non-strict에 따라 `lowerBound` vs `upperBound`를 골라 쓴다. (strict 증가 → lowerBound)
