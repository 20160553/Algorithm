## 문제 1. 보물 발굴 (비용 제한 탐색)

### 문제 요약

- `w`개의 열로 이루어진 땅에 보물이 단 하나 숨겨져 있다.
- `excavate(col)`을 호출하면 그 열을 파고 `depth[col]`만큼 **비용**이 발생한다.
- 반환값: 보물 발견 시 `0`, 보물이 왼쪽이면 `-1`, 오른쪽이면 `1`.
- 총비용이 `money`를 초과하지 않으면서 100% 확률로 보물 열을 찾아야 한다 (비용을 적게 써도 추가 점수 없음 → "초과만 안 하면 됨").
- 제한: `2 ≤ w ≤ 200`, `1 ≤ depth[i] ≤ 100,000`, `money ≥ 보장 탐색 최소 비용`.

---

### 내 풀이 분석

- **풀이 결과**: 실패 — `solution`/탐색 로직 미작성, `main`에 방향 메모만 존재.

**알고리즘**

작성된 코드는 없고, 주석으로 다음 직관만 기록했다.

> "비용 제한 있어서 이진 탐색 아님. 그리디도 아님. 이게 최적해가 된다고?"

**✅ 잘 짚은 것**

- **단순 이진 탐색이 정답이 아님**을 정확히 간파했다. 열마다 비용(`depth`)이 다르기 때문에, 항상 중앙을 파는 표준 이진 탐색은 "비싼 열을 자주 파게 되는" 최악의 경우 비용을 보장하지 못한다.
- **그리디도 아님**도 옳다. "지금 가장 싼 열부터" 같은 국소 선택은 이후 구간이 얼마나 비싸질지를 고려하지 못한다.

**❌ 문제점 / 개선점**

- 핵심 전환점("그럼 무엇인가?")에서 멈췄다. 정답은 **구간 DP(가중치 최적 이진 탐색 트리)** 이며, "각 구간에서 어느 열을 파야 최악 비용이 최소인가"를 미리 계산하는 문제다.
- "이게 최적해가 된다고?"의 답: 우리는 굴착 **결과(-1/0/1)를 고를 수 없으므로**, 어떤 열을 파든 **최악의 분기**(왼쪽·오른쪽 중 더 비싼 쪽)를 가정해 비용을 평가해야 한다. 이 미니맥스 구조 때문에 DP 점화식이 성립한다.

---

### 모범 답안: 구간 DP (미니맥스 / 최적 BST)

**알고리즘**

핵심은 "보물이 구간 `[lo,hi]` 안에 있다고 알려졌을 때, 100% 찾기 위한 **최악 비용**의 최솟값"을 `dp[lo][hi]`로 정의하는 것.

1. 구간 `[lo,hi]`에서 열 `c`를 파면 비용 `depth[c]`가 무조건 발생한다.
2. 결과가 `0`이면 끝, `-1`이면 `[lo,c-1]`, `+1`이면 `[c+1,hi]`로 좁혀진다.
3. 결과는 적이 정하므로 **최악**을 가정: `depth[c] + max(dp[lo][c-1], dp[c+1][hi])`.
4. 모든 `c`에 대해 최소가 되는 값이 `dp[lo][hi]`, 그때의 `c`를 `pick[lo][hi]`에 저장.
5. 구간 길이를 1부터 `w`까지 키워가며 채운다 (작은 구간이 먼저 필요).
6. 실제 탐색: `[0,w-1]`에서 시작해 `pick`이 가리키는 열을 `excavate`하고, 반환값에 따라 구간을 좁히며 트리를 따라 내려간다.

**점화식**

```
dp[lo][hi] = min over c in [lo,hi] of
             depth[c] + max( dp[lo][c-1], dp[c+1][hi] )
빈 구간(lo>hi) 비용 = 0
```

**시간복잡도**

| 단계 | 복잡도 |
|------|--------|
| 구간 길이 × 시작점 | O(w^2) |
| 각 구간의 분기 열 후보 c | O(w) |
| 실제 굴착 진행 | O(w) |
| 합계 | O(w^3) |

`w ≤ 200` → 약 8×10⁶ 연산으로 충분히 통과.

**공간복잡도**: O(w²) (`dp`, `pick` 테이블)

---

### 핵심 정리

- **"비용/가중치가 붙은 탐색"에서 표준 이진 탐색·그리디가 깨지면 → 구간 DP(최적 BST)를 의심하라.** 표준 이진 탐색은 "비교 비용이 모두 같다"는 가정 위에서만 최적이다.
- **결과를 내가 못 고르는 탐색 = 미니맥스.** 분기 결과를 통제할 수 없을 때는 항상 `max(왼쪽, 오른쪽)`로 최악을 잡아야 보장(100%)이 성립한다.
- **인터랙티브 문제 패턴**: 먼저 전략(어느 위치를 물을지)을 DP로 **오프라인 설계**한 뒤, 런타임에는 그 설계대로 질의(`excavate`)만 따라간다.
- 동일 계열: 최적 이진 탐색 트리(OBST), "추측 게임 최소 비용"(LeetCode Guess Number Higher or Lower II)이 정확히 같은 미니맥스 구간 DP다.

## 문제 2. 배달 로봇 최소 순회 비용 (TSP)

### 문제 요약

- 0번(창고)에서 출발 → 나머지 모든 지점을 **정확히 한 번씩** 방문 → 0번으로 복귀하는 최소 비용.
- `dist[i][j]` = i→j 직접 이동 비용, 길 없으면 `-1`. **방향성 있음**(`dist[i][j] ≠ dist[j][i]` 가능).
- 모든 지점을 방문하고 돌아오는 경로가 없으면 `-1` 반환.
- 제한: `2 ≤ n ≤ 16`, 간선 비용 `1 ~ 1,000,000` 또는 `-1`.
- 힌트가 명시: "방문 집합을 비트마스크로" → **외판원 순회(TSP) / Held-Karp**.

---

### 내 풀이 분석

- **풀이 결과**: 실패 (미완성) — DP 전이 루프(80~86줄)가 비어 있어 항상 `0` 반환.

**알고리즘**

`dp[방문 비트마스크][마지막 위치]`로 TSP를 풀려는 Held-Karp 방향은 정확히 잡았다. 하지만:
1. 앞에 Floyd-Warshall로 전구간 최단거리를 먼저 구함.
2. 기저값 `dp[1<<i][0] = dist[i][0]` 설정.
3. 본 전이 루프는 골격(`continue`)만 있고 비어 있음.

**시간복잡도**

- 의도한 DP: O(2^n · n^2)로 정확. 다만 미완성이라 정답을 산출하지 못함.

**✅ 잘 짚은 것**

- `n ≤ 16`, `15! ` 완전탐색은 TLE → **비트마스크 DP**라는 핵심 전환을 정확히 인지.
- 상태 정의 `dp[mask][last]`(방문 집합 + 마지막 위치)는 Held-Karp의 표준 정의 그대로다.

**❌ 문제점 / 개선점**

- **(개념 오류) Floyd-Warshall은 불필요하며 오히려 틀린다.** 각 지점을 "정확히 한 번"만 방문해야 하는데, 최단경로로 간선을 줄이면 **경유 지점을 두 번 방문**하는 셈이 된다. 직접 간선 `dist`만 써야 한다.
- **(버그) Floyd-Warshall 식이 틀림**: 67줄 `dist[i][mid] + dist[j][i]` → 올바른 식은 `dist[i][mid] + dist[mid][j]`.
- **(버그) `-1`(길 없음)을 INF로 처리하지 않음.** `Math.min(dist[i][j], ...)`에서 `dist[i][j] == -1`이면 잘못된 최솟값이 된다.
- **(미완성) 기저값이 거꾸로**: `dp[1<<i][0] = dist[i][0]`는 의미가 모호. 표준은 `dp[1][0] = 0`(0번만 방문, 현재 0).
- **(미완성) 전이 루프가 비어 있음**: `mask`에 새 지점 `next`를 추가하는 `dp[mask | (1<<next)][next] = min(..., dp[mask][last] + dist[last][next])` 전이가 없다.

---

### 모범 답안: Held-Karp (비트마스크 TSP DP)

**알고리즘**

상태: `dp[mask][last]` = 0에서 출발해 집합 `mask`의 지점을 모두 방문하고 현재 `last`에 있을 때의 최소 비용.

1. 초기화: 모든 값 INF, `dp[1][0] = 0` (0번만 방문, 현재 0).
2. 모든 `mask`에 대해(0번 포함된 것만), `mask`에 든 `last`에서 아직 안 든 `nx`로 확장:
   `dp[mask | (1<<nx)][nx] = min(..., dp[mask][last] + dist[last][nx])` (단 `dist[last][nx] ≠ -1`).
3. 전부 방문(`mask = FULL`)한 뒤, 마지막 지점 `last`에서 0번으로 복귀: `dp[FULL][last] + dist[last][0]` 중 최소.
4. 어떤 경로도 완성 못 하면(INF) `-1`.

**예시 추적** (예 1: 정답 80)

```
0 → 1 (10) → 3 (25) → 2 (30) → 0 (15) = 80
dp[{0,1}][1]=10 → dp[{0,1,3}][3]=35 → dp[{0,1,2,3}][2]=65 → +dist[2][0]=15 → 80
```

**시간복잡도**

| 단계 | 복잡도 |
|------|--------|
| 상태 수 (mask × last) | O(2^n · n) |
| 각 상태의 전이 (next) | O(n) |
| 합계 | O(2^n · n^2) |

`n=16` → 약 1.6×10⁷, 충분히 통과.

**공간복잡도**: O(2^n · n)

---

### 핵심 정리

- **`n ≤ 16~20` + "모든 정점 한 번씩" → 거의 확정적으로 비트마스크 TSP(Held-Karp).** `n!` 완전탐색이 막히면 `2^n` 상태로 압축.
- **상태 정의는 `dp[방문집합][현재위치]`** 가 표준. 시작점을 고정(여기선 0)하면 마지막에 복귀 비용만 더하면 된다.
- **"정확히 한 번 방문"하는 TSP에는 Floyd-Warshall을 쓰지 말 것** — 경유 최단경로는 중간 정점 재방문을 유발한다. 직접 간선만 사용.
- **`-1`(길 없음)은 반드시 INF로 치환**하고, 더하기 오버플로 방지를 위해 `INF = Integer.MAX_VALUE / 2` 같은 값을 쓴다.
