## 문제 1. 구내식당 식권 구매하기

### 문제 요약

- N일 동안 출근 일정(문자열 S, 1=출근/0=미출근)이 주어진다.
- 출근하는 날에는 반드시 식권으로 점심을 해결해야 한다.
- 식권 3종: 1일권(A, 산 날만), 3일권(B, 산 날부터 3일), 7일권(C, 산 날부터 7일).
- 기간 식권은 산 날을 포함한 연속 날짜에 적용되며, 미출근일이 포함돼도 기간은 그대로 흘러간다.
- 모든 출근일을 덮는 최소 비용을 구한다.
- 제한: 1 ≤ N ≤ 100,000, 1 ≤ A, B, C ≤ 100,000

---

### 내 풀이 분석

- **풀이 결과**: 부분 성공 — 예제는 통과하나 경계 케이스에서 오답

**알고리즘**

앞에서부터 채우는 2차원 DP.

- `dp[i][j]` = i일째가 종류 j 식권 구간의 **마지막 날**이 되도록 [1..i]를 타일링했을 때 최소 비용.
- 종류 j 식권은 길이 `period[j]`(1, 3, 7)이며, `i`에서 끝나면 `[i-period+1, i]`를 덮고 그 앞은 `dp[i-period][*]`로 이어 붙인다.
- 미출근일(`!flag`)에는 1일권 비용을 0으로 처리해 "식권 없이 그냥 넘기는 날"을 표현한다.
- 즉 전체 [1..N]을 길이 1/3/7 블록으로 **완전 분할**하는 최소 비용을 구하는 방식.

이 분할 관점 자체는 영리하다. 실제 최적해에서 식권이 겹칠 이유가 없으므로(겹침 = 낭비), 출근일을 덮는 비중첩 식권 + 미출근 공백을 0원 1일 블록으로 채우면 항상 분할로 환원되기 때문이다.

**시간복잡도**

- 상태 `O(N · 3)`, 전이마다 이전 종류 3개를 보므로 `O(N · 3 · 3) = O(N)`.

**✅ 잘 짚은 것**

- DP라는 큰 틀을 정확히 잡았다.
- 미출근일을 "비용 0짜리 1일 블록"으로 흡수한 아이디어가 깔끔하다.
- `INF`를 `long`으로 충분히 크게(최대 비용 ~10만×10만=100억 초과) 잡아 오버플로를 피했다.

**❌ 문제점 / 개선점**

- **경계 버그(핵심)**: 식권 구간이 반드시 N일 이내에서 끝나도록 `prev = i - period >= 0` 조건을 강제한다. 그래서 **막바지 출근일을 N 너머로 넘어가는 기간 식권으로 더 싸게 덮는 경우**를 표현하지 못한다.
  - 예) `N=1, A=5 B=3 C=1, S="1"` → 정답은 7일권 1원이지만, 사용자 코드는 7일권/3일권 구간이 `prev<0`으로 막혀 1일권 5원을 출력한다.
  - A < B, A < C 같은 가격 단조성이 보장되지 않으므로(제한상 C가 A보다 쌀 수 있음) 실제로 오답이 발생할 수 있다.
- **차원 낭비**: `dp[i][j]`에서 둘째 차원 j는 전이에 실제로 쓰이지 않는다(전이가 항상 `min_k dp[prev][k]`라 j별로 구분할 필요 없음). `dp[i]` 1차원으로 충분하다.
- 가독성: `works[i]` 초기화를 `true`로 깔고 0이면 false로 되돌리는 흐름보다 `str.charAt(i)=='1'` 한 줄이 명확하다.

---

### 모범 답안: 뒤에서부터 채우는 1차원 DP (greedy 시작점)

**알고리즘**

1. `dp[i]` = i일째부터 N일째까지의 모든 출근일을 해결하는 최소 비용으로 정의한다.
2. i가 미출근일이면 식권이 필요 없으므로 `dp[i] = dp[i+1]`.
3. i가 출근일이면 **i일을 덮는 식권은 i일에 사서 시작하는 것이 항상 최적**이다(앞쪽은 이미 처리됐으니 더 일찍 사면 손해). 따라서
   `dp[i] = min(A + dp[i+1], B + dp[i+3], C + dp[i+7])`.
4. `i+k > N`인 인덱스는 base case로 0 → 식권 구간이 N을 넘어가는 경우가 자동 허용된다.
5. 답은 `dp[1]`.

**예시 추적** (N=10, A=4000 B=9000 C=18000, S=1110011110)

```
출근일: 1,2,3 / (4,5 휴무) / 6,7,8,9 / (10 휴무)

1~3:  3일권 1장(day1) = 9000        (1일권 3장 12000보다 쌈)
6~9:  3일권(day6: 6,7,8) + 1일권(day9) = 9000 + 4000 = 13000
합계: 9000 + 13000 = 22000
```

**시간복잡도**

| 단계 | 복잡도 |
|------|--------|
| 입력 파싱 | O(N) |
| DP 채우기 | O(N) (각 i마다 상수 3가지 전이) |
| 합계 | O(N) |

**공간복잡도**: O(N)

---

### 핵심 정리

- "구간/기간으로 모든 지점을 덮는 최소 비용" 유형은 **첫(또는 마지막) 필수 지점을 기준으로 식권을 시작/끝내는 것이 최적**이라는 그리디 성질 위에 1차원 DP를 얹으면 깔끔하다.
- 뒤에서 채우는 DP는 `dp[i+k]`(미래 인덱스)가 N을 넘어가면 0이 되어, **구간이 끝을 넘어가는 식권**을 추가 처리 없이 자연스럽게 허용한다. 앞에서 채우는 DP는 이 경계를 따로 신경 써야 한다(이번 풀이의 버그 지점).
- DP 상태 차원을 늘리기 전에 "그 차원이 전이에 실제로 영향을 주는가?"를 먼저 점검하면 불필요한 `[i][j]`를 `[i]`로 줄일 수 있다.
- 비용 상한이 큰 문제(여기선 ~100억)는 `long`과 충분한 `INF`로 오버플로를 막는다.

## 문제 2. 택배 물류 분배

### 문제 요약

- 일렬로 놓인 N개의 보관함을 **순서를 유지한 채** M개의 연속 구간으로 나눈다(빈 구간 없음).
- 각 구간의 작업량 = 구간 내 box 합. 가장 큰 구간 합(= 최대 작업량)을 **최소화**한다.
- 그 "최대 작업량의 최솟값"을 출력한다.
- 제한: 1 ≤ M ≤ N ≤ 100,000, 1 ≤ box[i] ≤ 10,000 (합 최대 약 10억)

---

### 내 풀이 분석

- **풀이 결과**: 실패 (미완성) — `main`이 비어 있고 구현 코드가 없다.

**알고리즘**

작성된 것은 접근 메모뿐이다.

- 처음엔 `N-1 C M`(분할점 조합 전수) → 시간 초과를 정확히 인지했다.
- "누적합?", "이진탐색 + 누적합 응용인가?"까지 방향은 잡았으나 판정 로직으로 연결하지 못했다.

**✅ 잘 짚은 것**

- 완전탐색(`C(N-1, M-1)`)이 지수적으로 폭발해 불가능함을 정확히 판단했다.
- 파일 상단 힌트대로 **이분 탐색 + 그리디(파라메트릭 서치)** 라는 정답 방향을 인식하고 있었다.

**❌ 문제점 / 개선점**

- 구현 자체가 없다(미완성). 판정함수 `feasible(X)`를 코드로 옮기는 단계에서 막혔다.
- 누적합은 사실 **불필요**하다. 그리디 판정은 한 번 훑으며 합을 누적하면 되므로 별도 prefix sum 배열이 필요 없다. "누적합 응용"이라는 가정이 오히려 사고를 복잡하게 만들었다.
- 중복 `import java.io.IOException;`(60행)이 남아 있다 — 상단 `import java.io.*;`로 이미 포함된다.

---

### 모범 답안: 파라메트릭 서치 (이분 탐색 + 그리디 판정)

이 문제는 **"최댓값의 최소화"** 전형이며, Painters Partition / Book Allocation 으로 알려진 패턴이다.

**알고리즘**

1. 정답 X의 탐색 범위를 `[max(box), sum(box)]`로 잡는다.
   - 하한: 어떤 구간도 단일 보관함 하나는 담아야 하므로 최댓값보다 작을 수 없다.
   - 상한: 전부 한 구간이면 합이 곧 최대 작업량이다.
2. 판정 `feasible(X)`: 앞에서부터 한 구간 합이 X를 넘지 않게 그리디로 채우고, 넘으면 새 구간을 연다. 필요한 구간 수가 **M 이하**면 가능.
3. 가능하면 X를 더 줄이고(`hi = mid`), 불가능하면 키운다(`lo = mid + 1`). lower-bound 이분 탐색으로 최솟값을 찾는다.

**왜 그리디 판정이 맞는가**: 한도 X가 고정되면, 한 구간에 최대한 욱여넣는 것이 구간 수를 최소화한다. 구간 수가 작을수록 M 제약을 만족하기 쉬우므로, 그리디가 만든 최소 구간 수 ≤ M 이면 X는 달성 가능하다.

**예시 추적** (N=5, M=3, box = 1 2 3 4 5)

```
범위: [max=5, sum=15]
X=10 → [1 2 3 4][5] = 2구간 ≤ 3  가능 → hi=10
X=7  → [1 2 3][4][5] = 3구간 ≤ 3  가능 → hi=7
X=6  → [1 2 3][4][5] = 3구간 ≤ 3  가능 → hi=6
X=5  → [1 2][3][4][5] = 4구간 > 3 불가 → lo=6
lo == hi == 6  →  정답 6
```

**시간복잡도**

| 단계 | 복잡도 |
|------|--------|
| 입력 / 범위 계산 | O(N) |
| 이분 탐색 횟수 | O(log(sum)) |
| 각 판정(feasible) | O(N) |
| 합계 | O(N log(sum)) |

**공간복잡도**: O(N)

---

### 핵심 정리

- **"최댓값을 최소화하라 / 최솟값을 최대화하라"** 는 문구가 보이면 **답 자체를 이분 탐색**(파라메트릭 서치)하는 것을 1순위로 떠올린다.
- 핵심은 **단조 판정함수** 설계: "정답이 X 이하로 가능한가?"가 X에 대해 단조(가능→그 이상도 가능)여야 이분 탐색이 성립한다.
- 구간 분할 판정은 보통 **그리디 한 번 훑기 O(N)** 로 충분하다 — 누적합 배열이 항상 필요한 건 아니다.
- 탐색 범위의 하한/상한을 문제 의미로 정확히 잡는 것이 중요하다(여기선 `max(box)` ~ `sum(box)`).
