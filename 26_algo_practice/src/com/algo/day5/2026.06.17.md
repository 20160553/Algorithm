## 문제 1. 냉장고 정리 대작전

### 문제 요약

- 일렬로 놓인 N개의 음식을 **연속 구간** 여러 개로 나누어 전부 한 번씩 먹는다.
- 한 구간 `[l, r]`의 만족도 = `(구간 맛 점수 합) × (구간 내 최소 맛 점수)`.
- 먹는 횟수(구간 개수)는 **최대 K번**까지 허용된다.
- 전체 만족도 합의 최댓값을 구한다.
- 제한: `1 ≤ K ≤ N ≤ 500`, `1 ≤ A_i ≤ 1,000,000`. 정답은 64비트 정수 범위.

> ⚠️ 예시 오류: 예제 입력 `5 2 / 3 1 4 1 5`의 올바른 출력은 **34**이다 (제시된 45는 잘못된 값).
> K=2일 때 가능한 모든 2분할을 직접 확인해도 최댓값은 `[3,1,4,1] | [5] = 9×1 + 5×5 = 34`이며,
> sum×min / sum×max / 구간 수 변경 등 어떤 해석으로도 45는 나오지 않음(브루트포스로 확인).

---

### 내 풀이 분석

- **풀이 결과**: 실패 (미완성)

**알고리즘**

입력을 읽고 맛 점수 누적합 `tasteSum[]`까지만 구성한 단계에서 멈췄다.
main 주석에서 다음을 정확히 인지하고 있었다:
- 만족도 범위가 `long` 필요(`500 × 10^6 × 10^6`)
- 누적합으로 구간 합을 빠르게 구해야 함
- 완전탐색은 `500 C K`라 시간초과 — "구간을 더 적은 시간복잡도로 어떻게 나누지?"에서 막힘

**시간복잡도**

- 핵심 분할 로직 미구현이라 분석 불가.

**✅ 잘 짚은 것**

- `long` 오버플로 가능성을 미리 파악한 점.
- 구간 합 → 누적합 전처리로 연결한 점.
- 완전탐색이 `C(N, K)`로 폭발한다는 것을 스스로 인지한 점 (DP로 가는 길목).

**❌ 문제점 / 개선점**

- 가장 중요한 "구간 분할" 부분이 비어 있음.
- "구간의 끝점을 상태로 두는 DP"라는 한 발짝을 더 내딛지 못함.
- 구간 최소값을 어떻게 효율적으로 구할지에 대한 설계가 없음.

---

### 모범 답안: 구간 분할 DP (Partition DP)

**알고리즘**

1. `prefix[i]` = 앞 i개의 맛 점수 누적합.
2. `dp[j][i]` = 앞에서부터 i개의 음식을 **정확히 j개**의 연속 구간으로 나눴을 때 만족도 최댓값.
3. 전이: 마지막 구간을 `[p+1, i]`로 잡으면
   `dp[j][i] = max_{p} ( dp[j-1][p] + (prefix[i]-prefix[p]) × min(A[p+1..i]) )`.
4. `p`를 `i-1`부터 `j-1`까지 **거꾸로** 훑으면 마지막 구간의 최소값을 O(1)로 누적 갱신할 수 있어
   별도의 구간 최소값 전처리가 필요 없다.
5. "최대 K번"이므로 정답 = `max_{1 ≤ j ≤ K} dp[j][n]` (구간을 더 적게 쓰는 경우까지 모두 포함).

**예시 추적** (`5 2`, `3 1 4 1 5`)

```
j=1: 한 구간만 사용 → dp[1][5] = 14×1 = 14
j=2: 두 구간 분할 후보
     [3]|[1,4,1,5]      = 3×3 + 11×1 = 20
     [3,1]|[4,1,5]      = 4×1 + 10×1 = 14
     [3,1,4]|[1,5]      = 8×1 +  6×1 = 14
     [3,1,4,1]|[5]      = 9×1 +  5×5 = 34  ← 최댓값
정답 = max(14, 34) = 34
```

**시간복잡도**

| 단계 | 복잡도 |
|------|--------|
| 누적합 전처리 | O(N) |
| DP 전이 (j × i × p) | O(K · N^2) |
| 합계 | O(K · N^2) ≈ 1.25×10^8 |

**공간복잡도**: O(K · N)

---

### 핵심 정리

- "전체를 **연속 구간들로 분할**하고 각 구간의 점수 합을 최대/최소화" → 전형적인 **구간 분할 DP**.
  상태를 `dp[구간 수][끝점]`으로 두는 패턴을 기억하자.
- 완전탐색이 `C(N, K)`처럼 조합으로 터질 때, "구간의 **경계(끝점)**를 상태로 잡으면" 다항시간 DP가 되는 경우가 많다.
- 마지막 구간을 끝에서부터 늘려가며 훑으면 구간 합(누적합)·구간 최소/최대값을 **O(1) 누적 갱신**할 수 있어
  세그먼트 트리/희소 테이블 없이도 충분하다.
- "최대 K번"처럼 상한만 주어질 때는 `정확히 j개` DP를 만든 뒤 `j = 1..K`의 최댓값을 취하면 된다.
- 값의 곱이 큰 문제(`합 × 값`)는 항상 `long` 오버플로를 먼저 점검 (사용자가 잘 짚은 부분).

## 문제 2. 합이 K 이하인 가장 긴 연속 부분 수열

### 문제 요약

- 양의 정수 배열에서 **연속 부분 수열** 중 합이 `K 이하`인 것의 최대 길이를 구한다.
- 입력: 1번째 줄 `N K`, 2번째 줄 배열 원소 N개.
- 제한: `1 ≤ N ≤ 100,000`, `1 ≤ K ≤ 10^9`, `1 ≤ arr[i] ≤ 10,000`.

> ℹ️ 참고: 예제의 세 번째 줄 `0 1 3 6 10 15 16 17 19`는 실제 입력이 아니라
> 이해를 돕기 위한 **누적합 배열**이다. 실제 입력은 2줄(`N K` / 배열)뿐. 예제 출력 4는 정상.

---

### 내 풀이 분석

- **풀이 결과**: 부분 성공 (투 포인터 설계는 옳으나 입력 루프 off-by-one 버그)

**알고리즘**

누적합 배열 `sum[]`을 만들고 투 포인터(`l`, `r`)로 `sum[r]-sum[l] ≤ K`인 가장 긴 구간을 탐색.
조건을 만족하면 `r-l` 기록 후 `r++`, 초과하면 `l++`. 접근 방향 자체는 정확하다.

**시간복잡도**

- O(N) — `l`, `r`이 각각 최대 N번만 전진하므로 정상이라면 선형.

**✅ 잘 짚은 것**

- "모든 원소가 양수 → 투 포인터(슬라이딩 윈도우)" 라는 핵심 관찰을 정확히 함.
- 조건 위반 시 `l`을 당기고, 만족 시 `r`을 늘리는 단조성 기반 전개가 올바름.
- `l==r` 빈 윈도우 처리(`r++`)까지 신경 씀.

**❌ 문제점 / 개선점**

- **치명적 버그 (line 55)**: `for (int i = 1; i < n; i++)` → `i <= n` 이어야 함.
  마지막 원소를 읽지 않고 `sum[n]`이 0으로 남는다.
  - 예제는 우연히 4가 나오지만, **답이 마지막 원소를 포함하면 오답**.
    예) `N=3 K=100, arr=[1,1,100]` → 정답 2인데 사용자 코드는 3 출력.
- 누적합 배열은 불필요. 윈도우 합 변수 하나면 충분(인덱스 실수 여지 감소).
- `int` 누적합은 최대 `10^9`로 경계에 가까움 → `long` 권장.

---

### 모범 답안: 슬라이딩 윈도우 (투 포인터)

**알고리즘**

1. `right`를 0..N-1로 늘리며 `windowSum += arr[right]`.
2. 모든 원소가 양수이므로 `windowSum > K`이면 `left`를 당겨(`windowSum -= arr[left++]`) K 이하로 축소.
3. 매 단계 `answer = max(answer, right - left + 1)` 갱신.
4. 단조성 덕분에 한 번 지나간 `left`는 되돌아갈 필요가 없어 전체 O(N).

**시간복잡도**

| 단계 | 복잡도 |
|------|--------|
| 입력 | O(N) |
| 투 포인터 전개 | O(N) (left/right 각 최대 N회 이동) |
| 합계 | O(N) |

**공간복잡도**: O(N) (입력 저장), 윈도우 자체는 O(1)

---

### 핵심 정리

- "**양수 배열 + 연속 부분 수열 + 합 조건**" → 슬라이딩 윈도우 1순위. 합의 단조성이 핵심 전제.
  (음수가 섞이면 단조성이 깨져 투 포인터가 무너지므로 다른 접근 필요.)
- 누적합 배열을 채울 때 인덱스 범위(`1..N` 포함)를 항상 재확인 — `i < n` vs `i <= n` off-by-one 단골 버그.
- 예제만으로 정답을 확신하지 말 것. 이번처럼 버그가 있어도 특정 예제는 우연히 통과할 수 있다.
  → 경계 케이스(마지막 원소 포함 답, 전체 합이 K 초과 등)를 직접 만들어 검증하자.
