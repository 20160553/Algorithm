## 문제 1. 비밀번호 복구

### 문제 요약

- A, B, C로만 구성된 길이 N의 문자열 생성
- 금지 패턴(AAA, BBB, CCC, ABAB, BCBC, CACA)이 연속 부분 문자열로 등장하면 안 됨
- 사전순 가장 앞서는 유효 비밀번호 1개 출력, 불가능하면 -1 출력

---

### 내 풀이 분석

- **풀이 결과**: 부분 성공 (치명적 버그 존재, 로직 구조는 올바른 방향)

**알고리즘**

Deque를 이용한 DFS 탐색

1. 빈 문자열부터 시작해, C → B → A 순으로 addFirst하여 A가 먼저 탐색되도록 설계
2. 문자열 추가 후 금지 패턴 여부를 체크하여 유효한 경우만 큐에 추가
3. 길이 N에 도달한 첫 번째 문자열을 출력 (사전순 최소)

**시간복잡도**

- 가지치기가 없으면 O(3^N) 완탐
- 가지치기가 작동하면 실제 탐색 공간 대폭 감소

**✅ 잘 짚은 것**

- A를 마지막에 addFirst → 큐 앞에 A가 쌓임 → 사전순 최소 탐색 보장하는 구조
- 문자 추가마다 마지막 3~4글자만 검사하면 충분하다는 핵심 관찰

**❌ 문제점 / 개선점**

1. **치명적 버그**: `checkAvailability(convertToString(pw))`
   - `copy`에 문자를 추가했는데 `pw`(추가 전 문자열)로 유효성 검사
   - `pw`는 이미 유효한 상태이므로 항상 true 반환 → 가지치기 전혀 없음
   - 결과: N=3이면 첫 번째 탐색 완성 문자열이 "AAA"가 되어 오답 출력
   - 수정: `checkAvailability(convertToString(copy))`로 변경해야 함

2. **성능**: ArrayList<Character> + convertToString 조합
   - 매 단계마다 전체 리스트를 String으로 변환 (O(N) 복사)
   - StringBuilder 사용 시 append/deleteCharAt이 O(1) amortized

---

### 모범 답안 1: 재귀 DFS (백트래킹)

**알고리즘**

1. StringBuilder에 A, B, C 순으로 문자를 추가하며 DFS
2. 추가 후 마지막 3글자(AAA/BBB/CCC), 마지막 4글자(ABAB/BCBC/CACA) 검사
3. 금지 패턴이면 즉시 deleteCharAt으로 백트래킹
4. 길이 N 도달 시 그 문자열이 사전순 최솟값 (A 우선 탐색 보장)
5. 탐색 공간 전체 소진 시 null → "-1" 출력

**예시 추적** (N=5)

```
A → AA → AAA (AAA 금지 → 백트래킹)
          → AAB → AABA → AABAB (패턴 검사)
                 → AABAC ...
          → AAC → AACA (패턴 검사) ...
   → AB → ABA → ABAB (ABAB 금지 → 백트래킹)
                 → ABAC → ... (계속 탐색)
(첫 완성 문자열 = 정답)
```

**시간복잡도**

| 단계 | 복잡도 |
|------|--------|
| 최악 (금지 패턴 거의 없음) | O(3^N) |
| 실제 (가지치기 적용) | O(상태 수 × N) |

상태는 마지막 3글자에만 의존 → 반복 상태 수는 최대 27가지

**공간복잡도**: O(N) 재귀 스택 + StringBuilder

---

### 모범 답안 2: 반복문 DFS (명시적 스택)

**알고리즘**

1. 재귀 대신 `Deque`를 스택으로 사용 → 콜 스택 오버플로우 회피
2. 빈 문자열을 push하고 시작, pop한 상태에 C → B → A 순으로 문자를 붙여 push
3. 스택은 LIFO → C,B,A 순 push 시 A가 top → A 경로를 먼저 끝까지 탐색
4. 길이 N에 처음 도달한 문자열이 곧 사전순 최솟값
5. 스택이 빌 때까지 못 찾으면 -1 출력

**사전순 최소 보장 원리**

```
push 순서:  C → B → A   (A가 마지막에 들어가 top)
pop 순서:   A → B → C   (A가 먼저 나옴)
→ A로 시작하는 가지를 깊이 우선으로 끝까지 탐색
→ 첫 완성 문자열 = 사전순 최소
```

**재귀 버전과의 비교**

| 항목 | 재귀 DFS (모범 1) | 반복 DFS (모범 2) |
|------|------------------|------------------|
| 흐름 제어 | 콜 스택 (재귀) | 명시적 Deque 스택 |
| 사전순 보장 | for(A,B,C) 호출 순서 | push(C,B,A) → A가 top |
| 상태 관리 | StringBuilder 공유 + 백트래킹 | 부분 문자열을 스택에 독립 보관 |
| 공간복잡도 | O(N) | O(3^N) 최악 (동시 보관 문자열) |
| 장점 | 메모리 효율적 | 스택 오버플로우 없음, 구현 단순 |

**시간복잡도**: O(3^N) 최악, 가지치기로 실제 탐색 공간 대폭 감소

**공간복잡도**: O(3^N) 최악 (스택에 동시 보관되는 부분 문자열들)

---

### 핵심 정리

- "사전순 최소" 문자열 탐색 = DFS에서 A → B → C 순으로 탐색하면 자동 보장
- 금지 패턴 검사는 전체 문자열을 볼 필요 없이 "추가된 문자 기준 뒤 K글자"만 검사
- **유효성 검사 위치 중요**: 문자를 추가한 직후에 검사 (추가 전 검사는 의미 없음)
- ArrayList<Character> 대신 StringBuilder: append/deleteCharAt → 백트래킹 구현 관용구
- 상태가 마지막 K글자에만 의존하는 문제 = DP로도 풀 수 있음 (N이 큰 경우 필요)

## 문제 2. 보석 금고 경로

### 문제 요약

- N × N 격자, (0,0)에서 시작해 상하좌우로 이동하며 정확히 K개 칸 방문 (시작칸 포함)
- 같은 칸 재방문 금지, 장애물(`#`) 방문 금지
- 방문 칸 가치 합이 정확히 S여야 함
- 연속한 세 칸의 가치 a, b, c가 `a<b<c`(단조 증가) 또는 `a>b>c`(단조 감소)이면 금지
- 조건을 만족하는 **경로의 개수** 출력
- 제한: 3 ≤ N ≤ 8, 1 ≤ K ≤ 20, 0 ≤ S ≤ 100

---

### 내 풀이 분석

- **풀이 결과**: 성공 (정답 로직 완성)

**알고리즘**

DFS 백트래킹 완전탐색 (N ≤ 8이라 완탐 가능)

1. `(0,0)`에서 시작, `step`(방문 칸 수)을 1부터 세며 4방향 DFS
2. `step == k`일 때 `sum == s`면 정답 카운트
3. 방문 배열 `v`로 중복 방문 차단, 장애물(`map < 0`) 차단
4. 단조성 검사: 방문 값을 `StringBuilder`에 누적, 최근 3글자가 단조 증가/감소면 가지치기

**시간복잡도**

- 최악 O(4^K) (분기 4방향 × 깊이 K)
- N ≤ 8, 방문/장애물/단조 가지치기로 실제 탐색 공간 대폭 감소

**✅ 잘 짚은 것**

- "최소 거리가 아닌 경우의 수 + N ≤ 8 → 완탐" 정확한 판단
- 단조성을 "최근 3칸"만 보면 된다는 핵심 관찰 (시작값부터 누적해 첫 3칸도 검사)
- `step=1`부터 세고 `step==k`에서 종료 → **정확히 K칸** 방문 (칸 수 카운팅 정확)

**❌ 문제점 / 개선점**

1. **합 초과 가지치기 미구현**
   - main 주석에 "K칸 방문 안 했는데 S 초과 시 가지치기" 계획을 적었으나 코드엔 없음
   - 가치가 0 이상이므로 `sum > s`이면 더 진행해도 줄지 않음 → `if (sum > s) return;` 추가 가능
   - (제한이 작아 정답엔 무관, 성능 개선용)

2. **단조 검사를 문자열로 처리**
   - 매 호출 `sb.substring(...)`으로 문자열 생성 → 직전 칸 가치(`prev`)만 넘겨 직접 비교하면 더 간결
   - (값이 0~9 한 자리라 char 비교 = 숫자 비교로 정상 동작하긴 함)

> 💡 흔한 실수 2가지 (비교용)
> - **K+1칸 버그**: `depth=0`에서 시작해 `depth==K`로 종료하면 시작칸 포함 K+1칸을 방문하게 됨. "이동 횟수"가 아닌 "방문 칸 수"를 세야 함.
> - **첫 3칸 단조 누락**: 단조 검사를 "직전 2칸 이후"(`depth>=2`)부터 시작하면 시작칸 포함 첫 3칸을 빠뜨림. 직전 칸이 유효한 순간(2칸째)부터 검사해야 모든 연속 3칸을 본다.

---

### 모범 답안: DFS 백트래킹 (완전탐색 + 가지치기)

**알고리즘**

1. `(0,0)`에서 `cnt=1`(방문 칸 수)로 DFS 시작, 직전 칸 가치 `prev`를 함께 전달
2. `sum > s`면 즉시 가지치기 (가치 ≥ 0이라 초과분은 회복 불가)
3. `cnt == k`에서 `sum == s`면 정답 카운트
4. 4방향 이동 시 범위/방문/장애물 체크
5. 연속 3칸 `(prev, cur, val)`이 단조 증가/감소면 가지치기 (`prev >= 0`일 때만)
6. 재귀 시 `prev`에 현재 칸 가치를 넘겨 다음 단계의 "직전 칸"으로 사용

**핵심 포인트**

| 항목 | 처리 |
|------|------|
| 방문 칸 수 | `cnt=1` 시작, `cnt==k` 종료 → 정확히 K칸 |
| 단조 검사 범위 | `prev>=0`이면 검사 → 시작칸 포함 모든 연속 3칸 |
| 합 가지치기 | `sum>s` 즉시 컷 (가치 ≥ 0) |

**시간복잡도**

| 단계 | 복잡도 |
|------|--------|
| 최악 | O(4^K) |
| 실제 | 방문/장애물/단조/합 가지치기로 대폭 감소 |

**공간복잡도**: O(N²) 방문 배열 + O(K) 재귀 스택

---

### 핵심 정리

- "경로의 개수 + 작은 격자(N ≤ 8)" = DFS 백트래킹 완전탐색
- **칸 수 카운팅**: "방문 칸 수"인지 "이동 횟수"인지 명확히 — off-by-one 단골 버그
- **연속 K칸 조건**: 직전 1~2칸 상태만 인자로 넘기면 슬라이딩 윈도우식으로 검사 가능 (문자열 누적 불필요)
- 모든 연속 구간을 검사하려면 "검사 가능한 가장 이른 시점"부터 시작해야 첫 구간을 빠뜨리지 않음
- 가치가 음수가 없을 때만 "합 초과 → 가지치기"가 성립 (음수 있으면 회복 가능하므로 불가)
